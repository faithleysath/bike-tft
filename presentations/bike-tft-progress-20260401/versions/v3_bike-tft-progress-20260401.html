04
Dataset Fit

为什么这个数据集适合做需求预测基线

它天然有“起点需求 + 终点回流”的时空结构，也覆盖全年周期，但同时缺天气、容量和实时库存，这刚好决定了后续阶段的增强方向。

它能支持的问题

  • 从起点订单直接定义站点借车需求。
  • 从终点订单恢复站点回流压力。
  • 从全年数据学习日内、周内、季节周期。
  • 从经纬度学习站点空间差异。

它当前缺失的信息

  • 没有直接给天气和节假日特征。
  • 没有站点容量表和实时库存。
  • 没有 POI 或周边功能区信息。
  • 因此阶段 1 更适合作“基线”，不是最终论文版。

06
Preprocessing

预处理原理二：把聚合结果补成规则的站点小时面板

有了离站和到站统计后，还要把它们补成完整网格，并生成时序模型能直接消费的基础特征。

面板化怎么做

  • 先按总离站量筛出 top 200 活跃站点。
  • 构造完整的 all_ts × all_stations 网格。
  • 把没有观测订单的格子填成 0，而不是留空。
  • 合并站点经纬度，形成每个站点的静态属性。

派生了哪些基础特征

  • dep_count、arr_count、net_flow。
  • 小时、周几、月、week of year、是否周末。
  • 周期编码：hour_sin/cos、dow_sin/cos。
  • 连续时间索引 time_idx，方便后续切窗口。
07
Dataset Build

训练脚本原理一：样本数据集怎么构建

训练脚本不会直接把整张表丢给模型，而是先把面板转成 TimeSeriesDataSet，这样每个站点都能被切成 encoder / decoder 窗口。

字段如何分组

  • static_categoricals：station_id。
  • static_reals：station_lat、station_lng。
  • known_reals：小时、周几、月份、周期编码。
  • unknown_reals：dep_count、arr_count、net_flow 等历史统计。

窗口如何切

  • 目标列是 dep_count，分组键是 station_id。
  • max_encoder_length = 168，看过去 7 天。
  • max_prediction_length = 6，预测未来 6 小时。
  • 最后 168 小时作为验证区间。
08
Training Script

训练脚本原理二：模型如何构建并跑起来

这部分脚本把数据集定义接到 TFT + Lightning Trainer 上，负责训练、验证、checkpoint 和实验日志留档。

模型构建

  • 调用 TemporalFusionTransformer.from_dataset(train_ds) 自动对齐维度。
  • 代表性 run 用了 hidden_size=64、head=4、dropout=0.1。
  • 损失函数是 QuantileLoss(0.1, 0.5, 0.9)。
  • 支持 mixed precision，显式把 mask_bias 设为 -inf。

训练机制

  • 训练 / 验证各自构造 dataloader，验证 batch size 更大。
  • 启用 EarlyStopping、ModelCheckpoint、LR monitor。
  • 训练结束保存 best / last checkpoint。
  • 额外保存 timeseries_dataset_params.pt 便于复现实验。

Stage 02 Principle

增强思路一：先把天气和日历补进面板

先把城市级逐小时天气序列和节假日标签对齐到现有面板，这是补强未来可见信息最直接的一步。

怎么补天气 / 日历

  • 接入 NYC 区域逐小时历史天气序列。
  • 补温度、降雨、风速、湿度等变量。
  • 按本地时区和小时粒度对齐到现有面板。
  • 同步补节假日、工作日 / 周末等标签。

为什么这一块优先级高

  • 天气是共享单车需求最常见的外生冲击源。
  • 当前基线 decoder 侧主要只有日历类特征。
  • 补进天气后，远步长预测更有机会摆脱保守收缩。
  • 第一版即使采用城市级天气，也比完全缺失更合理。
14
Stage 02 Principle

增强思路二：用 trip 事件估计容量和库存代理

容量怎么估

  • 先统计每个站点每个时间点的 arrivals - departures。
  • 构造累计净流量曲线 S_t。
  • 利用 max(S_t) - min(S_t) 求容量下界。
  • 这给了站点物理约束的一个闭式近似。

库存代理怎么估

  • 给一个最小可行初始库存。
  • 按 inventory_t = inventory_(t-1) + arr_t - dep_t 递推。
  • 再裁剪到 [0, capacity_hat] 范围。
  • 得到“接近空桩 / 满桩”的状态代理。

容量下界的闭式思路

net_t = arrivals_t - departures_t
S_t = cumsum(net_t)
capacity_hat = max(S_t) - min(S_t)

库存代理的递推思路

inventory_t = clip(inventory_{t-1} + arr_t - dep_t, 0, capacity_hat)

它不是真实库存，但能把“站点接近空桩 / 满桩”的状态近似出来。
