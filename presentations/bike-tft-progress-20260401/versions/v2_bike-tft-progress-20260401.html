Dataset

我使用的 Citi Bike 数据集是什么

当前阶段用的是 Kaggle 上整理的 NYC Citi Bike trip-level 数据集，本质上是“逐单骑行记录”，而不是已经整理好的站点库存或需求面板。

12

13

Trip
每一行对应一次骑行订单
NYC
公开城市级共享单车运营数据

数据里最关键的信息

  • 时间：started_at、ended_at，决定借还车发生在哪个小时。
  • 站点：start_station_id、end_station_id，决定需求归属到哪个站点。
  • 空间：起终点经纬度可沉淀成站点静态位置特征。
  • 类型：rideable_type 和 member_casual 可以支持更细粒度分析。

为什么适合做毕设基线

  • 公开可复现，适合先把实验主线跑通。
  • 天然具备“起点需求 / 终点回流”的时空结构。
  • 覆盖全年，能学习日内、周内和季节模式。
  • 局限也很明确：原始数据没有直接提供天气、容量和实时库存。
阶段 1 的预处理脚本实际只先选了 10 个最关键字段来搭主线，包括时间、站点 ID、经纬度和车辆类型；像 member_casual、站点名称这类字段目前还没有充分利用。

Preprocessing

预处理脚本的核心原理：把 trip 记录变成规则的站点小时面板

原始数据是“不规则的事件流”，而 TFT 需要的是“规则、可分组、可切窗口”的时序面板，所以中间必须先做一次结构重建。

01

筛列与清洗

只读取关键列，解析时间戳，站点 ID 一律按字符串处理，避免像 4488.09 这类 ID 被数值化破坏。

02

双视角聚合

按 started_at 聚合离站需求 dep_count，按 ended_at 聚合到站回流 arr_count，并区分 classic / electric。

03

面板化补齐

筛出 top 200 活跃站点，构造完整的 all_ts × all_stations 网格，把没有观测到订单的格子补成 0。

04

特征工程

派生 net_flow、小时 / 周几 / 周末、周期 sin/cos 特征，并生成连续整数时间轴 time_idx。

事件流 → 规则面板 departure / arrival 双视角 完整时空网格补零 为 TFT 准备 time_idx
05
Dataset Build

训练脚本里，样本数据集是怎么构建的

训练脚本不是直接把整张表丢给模型，而是先把站点面板转成 TimeSeriesDataSet，让每个站点都能按滑动窗口切出 encoder / decoder 样本。

字段怎么分组输入模型

  • static_categoricals：station_id，让模型学习站点身份差异。
  • static_reals：station_lat、station_lng，作为站点静态位置特征。
  • time_varying_known_reals：小时、周几、月份、周末和周期 sin/cos 特征。
  • time_varying_unknown_reals：dep_count、arr_count、net_flow 以及 classic / electric 统计。

窗口样本怎么切出来

  • 目标列是 dep_count，分组键是 station_id。
  • max_encoder_length = 168，表示用过去 7 天做历史上下文。
  • max_prediction_length = 6，表示一次预测未来 6 个小时。
  • 最后 168 小时作为验证区间，training_cutoff = max_time_idx - validation_horizon。
  • 验证集通过 TimeSeriesDataSet.from_dataset(...) 从同一配置派生，保证编码方式一致。
这里还用了 GroupNormalizer(groups=["station_id"], transformation="softplus")，也就是每个站点按自己的尺度归一化目标值，减少不同站点量级差异带来的训练困难。
06
Training Script

模型怎么构建，训练怎么真正跑起来

这部分脚本把“数据集定义”接到 TemporalFusionTransformer 和 PyTorch Lightning Trainer 上，负责训练、验证、留档和可复现实验资产保存。

TFT 模型的实例化逻辑

  • 直接调用 TemporalFusionTransformer.from_dataset(train_ds)，让模型维度与数据集配置自动对齐。
  • 代表性 run 使用的关键超参是 hidden_size=64、hidden_continuous_size=32、attention_head_size=4、dropout=0.1。
  • 损失函数是 QuantileLoss([0.1, 0.5, 0.9])，不仅输出中位数预测，也保留不确定性区间。
  • mask_bias=-inf 是为了兼容 mixed precision，避免默认大负数在 float16 下溢出。

训练、验证和留档机制

  • 先把训练集和验证集转成 dataloader，验证 batch size 设置成训练的 2 倍。
  • Trainer 上挂了 EarlyStopping、LearningRateMonitor、ModelCheckpoint 和自定义 epoch timing callback。
  • 支持 precision、gradient_clip_val=0.1、accelerator="auto" 等训练参数。
  • 日志同时落到本地 CSVLogger，并可选上传到 LitLogger。
训练结束保存 best / last checkpoint 保留 timeseries_dataset_params.pt 后续报告脚本可以复用同一数据集配置

Stage 02 Principle

下一阶段增强数据集，核心是补天气并估计容量 / 库存代理

阶段 2 的目标不是“再多加几个字段”，而是把基线里缺失的外生冲击和物理约束补回来，让模型从纯时间模式走向更合理的机制解释。

天气和日历怎么补

  • 先接入 NYC 区域的逐小时历史天气序列，通过公开历史天气渠道拿到温度、降雨、风速、湿度等变量。
  • 按本地时区和小时粒度把天气表对齐到现有的 station × hour 面板，第一版可以先让所有站点共享同一份城市级天气序列。
  • 同时补齐节假日、工作日 / 周末等日历标签，让特殊日期不只靠时间索引隐含学习。
  • 如果后续能拿到天气预报，则可把天气作为未来可见特征；如果没有，就先作为历史辅助特征增强 encoder 侧信息。

容量和库存怎么预估

  • 从 trip 事件里先统计每个站点每个时间点的 arrivals - departures，构造累计净流量曲线。
  • 用累计曲线的最小相对库存和最大相对库存，求一个最小可行的站点容量下界。
  • 再结合估计的初始库存，把 arr_t - dep_t 累积起来并裁剪到容量范围内，得到库存代理序列。
  • 这份库存代理既能作为阶段 2 的历史状态特征，也能为阶段 4 的缺车 / 满桩识别和调度优化做准备。

容量下界的闭式思路

net_t = arrivals_t - departures_t
S_t = cumsum(net_t)
capacity_hat = max(S_t) - min(S_t)

库存代理的递推思路

inventory_t = clip(inventory_{t-1} + arr_t - dep_t, 0, capacity_hat)

它不是真实库存，但能把“站点接近空桩 / 满桩”的状态近似出来。
