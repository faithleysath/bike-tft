- 这个毕设到底在做什么
- 现在这个仓库做到哪一步了
- 为什么当前先用 Citi Bike
- 为什么先不做完整 OD

如果你过段时间回来已经忘了细节，就先把这份 README 从头到尾看一遍。

## 如果你现在很迷茫，先看这 7 句话

- 我的毕设要做的是一套“预测 + 调度 + 展示”的共享单车原型系统。
- 它的目标不是单纯拟合一个模型，而是提前判断未来哪些站点会缺车、哪些站点会车太多。
- 为了做到这一点，系统会结合历史流入、流出、库存等站点时序数据，以及天气、日历、POI、站点属性等信息来预测未来需求。
- 预测出来以后，还要进一步生成“从哪里搬车、搬到哪里、搬多少”的调度方案。
- 当前第一阶段先用公开的 Citi Bike 数据把最小链路跑通。
- 当前第一目标是站点级、小时级需求预测，不是完整 OD 预测。
- 当前最重要的事情不是把题目一次做满，而是确保毕设能顺利完成、能展示、能写论文。

## 一句话理解我的课题

一句话版本：

**让系统像一个有经验的共享单车调度员一样，提前知道未来哪里会缺车，然后提前把车调过去。**

再展开一点：

- 先整理历史站点状态数据，得到每个站点在每个小时的流入、流出、库存等信息
- 再结合天气、时间、节假日、POI、周边站点情况等信息
- 预测未来 1 到 24 小时每个站点的需求
- 然后根据预测结果和库存状态，生成再平衡调度方案
- 最后把预测结果和调度路线可视化展示出来

## 我的课题本质上在解决什么问题

共享单车运营里最常见的问题是：

- 某些站点在高峰期没车可借
- 某些站点在另一些时段车堆得太多
- 调度人员只能靠经验搬车，效率不稳定

所以这个课题想解决的是：

1. 先提前预测未来几个小时每个站点会不会忙
2. 再提前判断哪些站点可能缺车、哪些站点可能溢车
3. 再安排调度车去搬车，减少“无车可借”和“无桩可还”

你可以把整件事理解成一条业务链：

`数据整理 -> 需求预测 -> 调度决策 -> 可视化展示`

## 现在为什么先用 Citi Bike 公开数据

因为它能帮助我们先把最重要的主线跑通。

当前现实情况是：

- 校园单车数据还在制作中
- 毕设时间有限
- 公开数据能先验证方法是不是跑得通

所以现在的策略是：

- 先用 Citi Bike 完成一个最小可运行版本
- 等校园数据准备好后，再迁移到校园场景

这不是“退而求其次”，而是一个很合理的毕业设计推进方式。

## 当前公开数据里到底有什么

当前公开数据是 Citi Bike 的历史骑行订单数据。

它记录的是“每一笔骑行”而不是“每个站点每小时汇总结果”。

对课题本身来说，订单数据不是唯一必需输入。

只要能够拿到站点级的时间序列状态，比如：

- 每小时流入多少
- 每小时流出多少
- 当前库存大概多少
- 站点容量是多少

就已经可以支撑需求预测和调度决策。

当前之所以先用订单数据，是因为 Citi Bike 公开数据主要就是以订单形式提供的，我们需要先把它聚合成站点级时序表。

目前仓库已经检查过这批 CSV 的字段结构，12 个月文件的 schema 是一致的，核心字段有：

- `ride_id`：订单编号
- `rideable_type`：车辆类型，比如普通车、电助力车
- `started_at`：借车时间
- `ended_at`：还车时间
- `start_station_name` / `start_station_id`：借车站点
- `end_station_name` / `end_station_id`：还车站点
- `start_lat` / `start_lng`：借车站点经纬度
- `end_lat` / `end_lng`：还车站点经纬度
- `member_casual`：用户类型，会员或临时用户

这些信息足够我们先做出：

- 每个站点每小时借出多少车
- 每个站点每小时还回多少车
- 每个站点的净流入 / 净流出
- 车辆类型和用户类型的拆分统计

但这份公开数据本身还没有直接提供：

- 天气
- POI
- 站点容量
- 实时库存
- 节假日 / 校历 / 活动信息

这些都需要后面额外补充。

## 这份毕设不是在做什么

为了避免自己后面越做越散，这里明确几个“当前阶段先不做”的点。

当前阶段，这个课题不是：

- 不是先做一个特别复杂的完整生产系统
- 不是先做完整城市级实时部署
- 不是先做全量 OD 矩阵预测
- 不是先做最先进最复杂的图神经网络体系
- 不是一开始就把所有特征和模块全部堆上去

当前阶段最重要的是：

**先把一个能跑、能讲、能展示、能写进论文的主线版本做出来。**

## 为什么现在先不做完整 OD

这里专门写清楚，因为这件事你后面大概率还会反复犹豫。

原始订单数据确实包含更丰富的信息：

- A 站这个小时出去多少车
- 这些车分别去了哪些站
- A 站到 B 站的流量是多少

这些信息当然有价值。

但是当前毕设主线里，第一阶段不把“完整 OD 预测”当主任务，原因很简单：

- OD 维度非常高，站点一多就会爆炸
- OD 数据很稀疏，很多站点对在大多数时段都是 0
- 一上来做 OD 会让问题复杂很多，容易导致毕设做不完
- 你当前真正需要先解决的问题，是“每个站点未来会不会缺车”

对于“站点会不会缺车”这个目标来说，第一版更重要的是：

- 每站每小时借出多少
- 每站每小时还回多少
- 当前库存大概多少
- 站点容量是多少

所以当前阶段的正确策略是：

- 先做站点级小时需求预测
- 先把 `进 / 出 / 净流` 跑通
- 如果后面还有时间，再把 OD 压缩成“网络效应特征”加入模型

换句话说：

**OD 不是没用，而是更适合作为第二阶段增强项，而不是第一阶段主任务。**

## TFT 用人话怎么理解

TFT 不需要一开始就死磕公式。

你可以先把它理解成：

**一个擅长看时间序列、能同时利用过去信息和已知未来信息的高级预测模型。**

在这个课题里，它的价值主要有三点：

- 能同时看很多特征，比如时间、天气、站点属性、历史需求
- 能一次预测未来多个时间步
- 能输出不同分位数，给出风险区间，而不是只给一个单点值

你现在只需要知道：

- `0.1` 分位数：偏低、偏保守的需求估计
- `0.5` 分位数：中间值，可以当成典型预测
- `0.9` 分位数：偏高、偏忙情况下的需求估计

这对调度很有用，因为调度不能只看“一个平均值”，还要考虑风险。

## 调度模块用人话怎么理解

调度模块就是在回答这个问题：

**既然已经预测出未来哪里缺车、哪里车太多，那应该怎么搬车最划算？**

它大概会做这些事：

1. 根据预测结果和当前库存，找出缺车站和溢车站
2. 形成一批调度任务
3. 决定从哪些站取车、往哪些站送车、每次送多少
4. 尽量让调度距离更短、成本更低、缺车率更低

你可以把 OR-Tools 理解成：

**帮你算调度路径和任务分配的工具。**

## 当前公开数据能支持什么，不能支持什么

### 当前已经能支持

- 站点级小时需求统计
- 基础时间特征
- 基于订单的进出流量统计
- 站点位置相关特征的准备

### 当前还不能直接支持

- 真实实时库存

这说明：

**Citi Bike 足够让你先把“预测主线”跑起来，但要完全贴合最终题目，还需要逐步补更多数据源。**

建议按下面这个顺序往前推：

1. 跑通 Citi Bike 最小版本

3. 补 POI 和站点静态特征

- 公开数据实验能完整复现

- 至少有几个基线模型
- 至少有一个可运行的调度模块

## 等校园数据准备好以后怎么迁移

后面接校园数据时，不需要把整个仓库推倒重来。

更合理的做法是：

- 保留当前这条主线
- 把校园数据整理成类似的“站点 x 时间”结构
- 把天气、校历、POI、站点容量等按相同思路并进来
- 尽量复用现有训练和评估逻辑

如果校园数据字段和 Citi Bike 不完全一样，也没关系。

真正要对齐的不是字段名本身，而是数据结构思路：

- 每行表示一个站点在某个时间片的状态
- 有目标值
- 有时间索引
- 有静态特征
- 有过去特征
- 有未来已知特征
